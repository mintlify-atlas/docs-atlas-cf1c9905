This guide walks you through cloning and running all three services — Edge Service, Instrument Service, and Chat Service — on your local machine with two fully isolated tenants.

## Prerequisites

Before you begin, make sure you have the following installed:

- **Java 21** — all three services use the Java 21 toolchain
- **Docker** and **Docker Compose** — required to run Keycloak, PostgreSQL, and the observability stack
- **Gradle** (or use the included `./gradlew` wrapper) — used to build and run each service
- **Ollama** _(optional)_ — only required if you want to run Chat Service without Testcontainers; install from [ollama.ai](https://ollama.ai)

---

## Get up and running

<Steps>

<Step title="Clone the repository">

Clone the project from GitHub and navigate into it:

```bash
git clone https://github.com/ThomasVitale/spring-boot-multitenancy.git
cd spring-boot-multitenancy
```

</Step>

<Step title="Configure tenant DNS">

The two tenants — `dukes` and `beans` — are resolved by hostname. Add the following entries to your `/etc/hosts` file so that `dukes.rock` and `beans.rock` resolve to localhost:

```bash
127.0.0.1       dukes.rock
127.0.0.1       beans.rock
```

<Note>
On macOS and Linux, `/etc/hosts` requires `sudo` to edit. On Windows, the equivalent file is `C:\Windows\System32\drivers\etc\hosts`.
</Note>

</Step>

<Step title="Start backing services with Docker Compose">

From the project root, start all infrastructure services in detached mode:

```bash
docker-compose up -d
```

This starts the following containers:

| Service | Purpose | Port |
|---|---|---|
| **Keycloak** | Authentication and tenant realm management | `8080` |
| **PostgreSQL** | Grafana metadata database | — |
| **Chroma** | Vector database for Chat Service | `8000` |
| **Grafana** | Observability dashboards | `3000` |
| **Loki** | Log aggregation | `3100` |
| **Tempo** | Distributed tracing | `4317`, `4318` |
| **Prometheus** | Metrics collection | — |
| **Alloy** | Grafana telemetry collector (OTLP receiver) | `4317`, `4318`, `12345` |

<Tip>
Once running, you can access Grafana at [http://localhost:3000](http://localhost:3000) using `user` / `password`.
</Tip>

</Step>

<Step title="Run Instrument Service">

Instrument Service (port `8181`) uses Spring Data JPA with Flyway migrations and a multi-schema PostgreSQL setup — one schema per tenant (`DUKES` and `BEANS`). Use `bootTestRun` to let Testcontainers automatically provision a PostgreSQL instance:

```bash
./gradlew :instrument-service:bootTestRun
```

<Note>
`bootTestRun` relies on Testcontainers to spin up a dedicated PostgreSQL container, so you do not need a separately configured database connection for this service.
</Note>

</Step>

<Step title="Run Edge Service">

Edge Service (port `80`) acts as the API gateway. It routes requests to Instrument Service based on the `Host` header (e.g. `dukes.rock` → tenant `dukes`) and relays OAuth2 tokens via the `TokenRelay` filter. Two tenants are active: `dukes` and `beans`, each backed by a separate Keycloak realm.

```bash
./gradlew :edge-service:bootRun
```

</Step>

<Step title="Run Chat Service">

Chat Service (port `8282`) uses Spring AI with Ollama (llama2 model) and Chroma for vector storage. You have two options:

**Option A — Testcontainers (recommended for local dev):**

```bash
./gradlew :chat-service:bootTestRun
```

**Option B — Local Ollama:**

If you have Ollama installed and prefer to run the model directly, pull and start the llama2 model first:

```bash
ollama run llama2
```

Then start Chat Service normally:

```bash
./gradlew :chat-service:bootRun
```

<Note>
Chat Service uses `llama2` for both chat completions and embeddings, as configured in `chat-service/src/main/resources/application.yml`.
</Note>

</Step>

<Step title="Test the two tenants">

With all three services running, open your browser and verify each tenant sees its own isolated data.

**Tenant: dukes**

1. Navigate to [http://dukes.rock/instruments/](http://dukes.rock/instruments/)
2. You are redirected to Keycloak for the `dukes` realm
3. Log in with `isabelle` / `password`
4. You should see the list of instruments belonging to the Dukes rock band

**Tenant: beans**

1. Open a second browser window (or a private/incognito window) and navigate to [http://beans.rock/instruments/](http://beans.rock/instruments/)
2. You are redirected to Keycloak for the `beans` realm
3. Log in with `bjorn` / `password`
4. You should see the list of instruments belonging to the Beans rock band

<Tip>
Use separate browser profiles or incognito windows to hold sessions for both tenants simultaneously without conflicts.
</Tip>

</Step>

</Steps>

## How tenant isolation works

The Edge Service reads the subdomain from the `Host` header using the route predicate `Host={tenant}.rock` and injects it as the `X-TenantId` request header downstream. Each tenant maps to its own Keycloak realm for authentication and its own PostgreSQL schema in Instrument Service — `DUKES` and `BEANS` — ensuring complete data isolation between tenants.

A third tenant, `trixie`, is defined in the configuration but set to `enabled: false`, demonstrating how tenants can be toggled without code changes.
